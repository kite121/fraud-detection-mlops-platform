Goal:
- Verify that ingesting a degraded batch automatically triggers retraining, creates a retraining job, and runs it in worker-service.

Preconditions:
- The platform is running in Docker.
- A current best model exists and can be used as a reference for drift comparison.
- Auto-monitoring listener is active in ingestion-service.
- A deliberately drifted dataset file was prepared as degraded_batch.csv.

Steps performed:
1. Uploaded degraded_batch.csv through POST /ingest.
2. Checked ingestion-service logs and confirmed that drift exceeded threshold and a retraining job was queued automatically.
3. Checked worker-service logs, PostgreSQL, and GET /jobs/{job_id} for the retraining job lifecycle and result.

Expected result:
- The degraded batch is ingested successfully.
- Auto-monitoring detects drift and queues a retraining job automatically.
- Worker-service executes retraining successfully.
- A retraining record is present in training_jobs with the new model version and MLflow run id.

- PostgreSQL confirmed a retraining job record with status completed.
- GET /jobs/train_fcee91c3099e returned completed with model_version v003 and a valid mlflow_run_id.
