The POST /ingest endpoint is implemented as a thin orchestration layer that reuses the existing service modules instead of duplicating their logic. This design keeps responsibilities separated: validation.py checks CSV correctness, storage.py uploads the raw file to MinIO, and metadata.py stores batch metadata in PostgreSQL.

The ingestion flow accepts a CSV file together with client_id and dataset_version, saves the uploaded file temporarily, validates it, uploads it to shared object storage under a client-specific prefix, and then writes a metadata record for that batch into the database. The endpoint returns a compact success response with batch_id, rows_count, and storage_path, while invalid files are rejected with an HTTP error before any storage or database write happens.
